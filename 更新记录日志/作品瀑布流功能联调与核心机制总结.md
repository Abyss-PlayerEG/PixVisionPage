# 作品瀑布流功能联调与核心机制总结

## 1. 概述
本文档记录了 PixVisionPage 项目中“作品瀑布流”功能的完整实现流程、前后端联调规范以及核心的视觉计算算法。该功能旨在通过水平滚动的交互形式，展示高质量的作品缩略图，并采用了智能高度分配策略以优化视觉体验。

## 2. 技术架构 (三层架构)
遵循项目统一的前后端联调规范，瀑布流功能严格划分为以下三层：

### 2.1 API 层 (`src/api/workApi.js`)
- **职责**：负责 HTTP 请求发送、响应标准化处理及数据格式转换。
- **核心函数**：
  - `fetchWorkPage(current, size)`：获取分页作品数据，兼容后端 `code`/`recode` 状态码。
  - `transformWorksToWaterfallFormat(records)`：将原始作品记录转换为瀑布流组件所需的 `{src, alt, height}` 格式，并在此阶段执行智能高度预计算。

### 2.2 Composable 层 (`src/composables/useWorkWaterfall.js`)
- **职责**：管理瀑布流的业务逻辑、加载状态及错误处理。
- **核心逻辑**：
  - 调用 API 层获取数据。
  - 维护 `waterfallImages`（图片数据）、`isLoading`（加载状态）和 `error`（错误信息）。
  - 在组件挂载时自动触发第一页数据的加载。

### 2.3 视图层 (`src/views/mainIndex.vue` & `src/components/Waterfall.vue`)
- **职责**：负责 UI 渲染、GSAP 动画绑定及用户交互。
- **核心组件**：
  - `Waterfall.vue`：实现水平滚动、列动态计算、骨架屏展示及 ScrollTrigger 联动。

## 3. 核心计算机制 (视觉算法)

为了解决传统瀑布流高度随机导致的视觉杂乱问题，我们设计了一套**“智能高度分配 + 周期性交换 + 留白补偿”**的组合算法。

### 3.1 基础高度随机池
- **档位设置**：`[260, 310, 360, 410, 460]`。
- **目的**：通过 5 个等差数列的高度档位，在保证错落感的同时维持视觉上的协调性。

### 3.2 列尾自适应逻辑
- **触发条件**：当模拟列的剩余空间小于 `350px` 时。
- **处理方式**：不再从随机池中选取，而是直接将当前图片高度设为**剩余空间大小**（最小保底 `100px`），确保列内不留大片空白。

### 3.3 周期性高度交换机制
为了打破随机生成的规律感，在数据生成后对每一列执行基于索引的交换策略：
- **第 1, 4, 7... 列 (索引 % 3 === 0)**：保持原样，不交换。
- **第 2, 5, 8... 列 (索引 % 3 === 1)**：交换**最后一张**与**倒数第二张**的高度。
- **第 3, 6, 9... 列 (索引 % 3 === 2)**：交换**最后一张**与**第一张**的高度。

### 3.4 最后一列留白填充
- **逻辑**：在所有交换完成后，单独计算最后一列的总高度。如果存在留白，将所有剩余高度累加到该列的**最后一张图片**上。
- **效果**：确保瀑布流的最右侧边缘与容器底部完美对齐，消除视觉缺口。

## 4. 前后端联调规范

### 4.1 接口配置
- **配置文件**：`src/config/api.js`
- **基础地址**：`API_BASE_URL = 'http://172.18.196.178:9090'`
- **作品接口**：`/api/work/page/{current}/{size}`

### 4.2 响应处理标准
- **状态码兼容**：统一处理 `result.code` 或 `result.recode`。
- **标准化返回**：所有 API 函数均返回 `{ success: boolean, data?: any, message?: string }`。
- **URL 编码**：涉及用户输入的参数（如邮箱、用户名）在发送前必须进行 `encodeURIComponent` 处理。

### 4.3 调试与日志
- **关键节点打印**：在 API 层保留请求参数和响应结果的 `console.log`，方便排查网络问题。
- **错误捕获**：使用 `try-catch` 包裹所有 `fetch` 请求，确保网络异常时能返回友好的错误提示。

## 5. 性能与优化

### 5.1 GSAP ScrollTrigger 优化
- **跟手性**：设置 `scrub: 1` 实现完全跟手滚动，消除回弹延迟。
- **防抖动**：开启 `fastScrollEnd: true`，确保快速滚动后立即到达最终位置。
- **内存清理**：在 `onUnmounted` 钩子中严格调用 `trigger.kill()` 和 `ScrollTrigger.clearScrollMemory()`，防止内存泄漏。

### 5.2 布局稳定性
- **骨架屏**：在数据加载完成前展示具有闪烁动画的占位块，提升用户体验。
- **列宽固定**：每列宽度固定为 `300px`，配合 `flex-shrink: 0` 确保水平滚动时的布局稳定性。

## 6. 常见问题排查 (FAQ)

| 问题现象 | 可能原因 | 解决方案 |
| :--- | :--- | :--- |
| 图片被裁剪 | 容器设置了 `overflow: hidden` | 检查 `.waterfall-gallery` 的 CSS 样式，确保高度计算正确 |
| 滚动位置偏移 | 上方内容未完全渲染 | 增加 `setTimeout` 延迟初始化 ScrollTrigger，并调用全局 `refresh()` |
| 最后一列留白 | 数据量不足以填满最后一列 | 检查 `transformWorksToWaterfallFormat` 中的留白补偿逻辑是否生效 |
| 状态码不匹配 | 后端返回字段名变更 | 确认 API 层是否正确兼容了 `code` 和 `recode` 字段 |

## 7. 后续迭代建议
1. **真实宽高比适配**：目前高度为模拟值，未来可接入后端返回的图片真实宽高比进行更精准的布局。
2. **无限滚动加载**：当前为单次加载，后续可增加触底（或触右）自动加载下一页的功能。
3. **图片懒加载优化**：结合 `IntersectionObserver` 进一步优化大量图片进入视口时的加载性能。
