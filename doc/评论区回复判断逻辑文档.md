# 评论区回复判断逻辑文档

## 1. 核心判断逻辑

**判断条件**（`WorkDetail.vue` L151）：
```vue
<span v-if="reply.replied_nickname && reply.parent_comment_id !== reply.in_comment_id">
  @{{ reply.replied_nickname }}
</span>
```

## 2. 两种回复场景

| 场景 | 条件 | 显示效果 | API参数 |
|------|------|----------|---------|
| **回复一级评论** | `parent_comment_id === in_comment_id` | 不显示@ | `parentCommentId`: 一级评论ID, `repliedUserId`: 不传 |
| **回复二级评论** | `parent_comment_id !== in_comment_id` | 显示@被回复者 | `parentCommentId`: 被回复二级评论ID, `repliedUserId`: 被回复者ID |

## 3. 数据结构说明

### 一级评论字段
- `comment_id`：评论ID
- `in_comment_id`：所属一级评论ID（一级评论的此字段等于自身comment_id）
- `parent_comment_id`：父评论ID（一级评论的此字段等于自身comment_id）
- `children`：二级评论数组

### 二级评论字段
- `comment_id`：评论ID
- `parent_comment_id`：直接回复的评论ID（可能是二级评论）
- `in_comment_id`：所属一级评论ID
- `replied_nickname`：被回复者昵称
- `replied_user_id`：被回复者用户ID

## 4. 回复流程实现

### 发起回复（`useWorkDetail.js` L176）
```javascript
const startReply = (comment, parentId, userId) => {
  replyingTo.value = {
    commentId: comment.comment_id,
    parentId: parentId !== undefined ? parentId : comment.comment_id,
    nickname: comment.nickname || '匿名用户',
    userId: userId,
  }
}
```

### 提交回复（`useWorkDetail.js` L192）
```javascript
const result = await addComment({
  workId: id,
  commentFloor: 2,
  commentText: text,
  parentCommentId: replyingTo.value.parentId,
  repliedUserId: replyingTo.value.userId || undefined,
})
```

## 5. 模板调用示例

<button @click="startReply(comment)">回复</button>

### 二级评论的回复按钮（`WorkDetail.vue` L156）

## 6. 后端接口定义

### 6.1 新增评论接口

**接口地址**：`POST /api/comment/add`

**认证方式**：
- Header: `Authorization: Bearer <token>`
- 或 URL 参数: `?token=<token>`

**请求参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `workId` | Integer | 是 | 作品 ID |
| `commentFloor` | String | 是 | 评论层级：`1` = 一级评论，`2` = 二级评论 |
| `commentText` | String | 是 | 评论内容，最多 125 个汉字 |
| `parentCommentId` | Integer | 二级评论必填 | 父评论 ID |
| `repliedUserId` | Integer | 否 | 被回复者用户 ID（由后端自动填充） |

**响应示例**：
- **审核通过**：`{ "data": true, "message": "评论新增成功" }`
- **AI 审核不通过**：`{ "data": false, "message": "违规内容：{原因}" }`
- **AI 审核存疑**：`{ "data": true, "message": "评论成功，等待人工审核" }`

**审核状态码**：
- `10`：正常（审核通过）
- `20`：待审核
- `30`：未过审

**业务逻辑**：
1. 验证 Token 有效性
2. 校验参数完整性
3. 验证评论层级（必须为 1 或 2）
4. 验证评论内容长度（≤125 字）
5. 二级评论必须提供 `parentCommentId`
6. 一级评论不应提供 `parentCommentId`
7. 验证父评论存在且属于同一作品
8. 调用 AI 审核服务
9. 创建评论并返回结果

### 6.3 删除评论接口

**接口地址**：`POST /api/comment/delete`

**请求参数**：
- `commentId`：评论 ID

**限制**：仅可删除自己的评论

## 7. 判断逻辑图示

```
回复一级评论：
├── 点击一级评论的"回复"按钮
├── startReply(comment) → parentId = comment.comment_id
├── 提交时：parentCommentId = 一级评论ID, repliedUserId = undefined
└── 显示时不带@（因为 parent_comment_id === in_comment_id）

回复二级评论：
├── 点击二级评论的"回复"按钮  
├── startReply(reply, reply.comment_id, reply.replied_user_id)
├── 提交时：parentCommentId = 二级评论ID, repliedUserId = 被回复者ID
└── 显示时带@（因为 parent_comment_id !== in_comment_id）
```

## 9. 总结

评论区回复功能通过以下机制实现：
1. **数据结构**：一级评论包含`children`数组存储二级评论
2. **判断逻辑**：比较`parent_comment_id`与`in_comment_id`是否相等
3. **显示规则**：相等时不显示@（回复一级评论），不等时显示@（回复二级评论）
4. **API参数**：根据回复目标传递不同的`parentCommentId`和`repliedUserId`
